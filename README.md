# Django Custom Auth

Данный проект представляет собой реализацию кастомной системы аутентификации и авторизации на базе Django, выполненную в соответствии с техническим заданием.

## Технологический стек
- **Framework:** Django 6.0+
- **Security:** bcrypt (хэширование паролей)
- **Database:** SQLite (развернута схема с Role, User, Session, Article, AccessRule)
- **Auth:** Кастомная Middleware + Session Management через Cookies

## Описание системы разграничения прав доступа

В соответствии с требованиями ТЗ, система не использует встроенные механизмы `django.contrib.auth`. Реализована собственная структура:

### 1. Модель данных
Система управления доступом строится на трех ключевых сущностях:
- **Role (Роли):** Определяет группу пользователя (например, Admin, User, Guest).
- **AccessRule (Правила доступа):** Связующая таблица, которая определяет, какие действия разрешены конкретной роли для конкретного элемента системы.
- **User (Пользователи):** Каждый пользователь привязан к одной роли.

**Поля правил (AccessRule):**
- `element_name`: Название объекта (например, "Article").
- `create_permission`: Право на создание.
- `delete_own_permission`: Право на удаление только своих объектов.
- `delete_all_permission`: Право на удаление любых объектов в системе.

### 2. Механизм аутентификации
- **Middleware (`CustomAuthMiddleware`):** Перехватывает каждый запрос, считывает `sessionid` из Cookies и сопоставляет его с активной сессией в БД. Если сессия валидна и не истекла, объект пользователя прикрепляется к `request.user`.
- **Сессии:** Хранятся в таблице `Session` с указанием времени истечения (`expire_at`).

### 3. Обработка прав в логике (Views)
В каждой защищенной функции реализована проверка:
1. **Идентификация:** Если `request.user` не определен - возвращается `401 Unauthorized`.
2. **Авторизация:** Проверяется наличие записи в `AccessRule` для роли текущего пользователя. Если действие запрещено - возвращается `403 Forbidden`.

## Основные возможности
- **Регистрация:** С проверкой уникальности Email и подтверждением пароля. Пароли хранятся в виде защищенных хэшей (bcrypt).
- **Мягкое удаление:** При удалении аккаунта сессия аннулируется, а пользователю выставляется флаг `is_active=False`. Он больше не может войти, но данные сохраняются в БД.
- **Управление статьями:** Удаление статьи требует либо владения ею (при наличии права `delete_own`), либо глобального права `delete_all`.
- **Admin API (`/manage_rules/`):** Специальный интерфейс для администраторов, позволяющий просматривать и изменять права доступа для любых ролей в реальном времени.
